# ADR 0001 — 텍스트 파이프라인 → 멀티미디어(쇼츠+이미지) 파이프라인 전환

- **상태**: 채택 (2026-07-18)
- **맥락**: 원형 `py-msa-ai`는 텍스트 LLM 중심. CONVEY의 산출물은 텍스트가 아니라 **쇼츠 영상 + 이미지**(둘 다 동등)로 확정됨(설계 문답 Q3b). 무거운 로컬 GPU 미디어 서비스를 둘지, 외부 API를 쓸지가 쟁점.
- **결정**: 멀티미디어 파이프라인으로 전환하되 **미디어 생성(이미지·TTS·영상클립)은 외부 API**로, **영상합성은 로컬 ffmpeg**로 한다. 미디어 서비스는 `market-feed`의 부패방지 계층 패턴을 재사용한 **가벼운 외부 API 래퍼 워커**(`image-gen`·`tts`·`video-clip`)와 로컬 합성 워커(`video-assembly`)로 추가.
- **트레이드오프**: 외부 API 의존·비용·레이트리밋 발생 vs 로컬 GPU 운영 부담 회피. → 부패방지 계층으로 격리, 원문 텍스트 전체는 외부로 넘기지 않음(프롬프트·에셋만).
- **영향**: 원형 골격(MSA·게이트웨이·비동기 Kafka·서비스별 DB·워커)은 **그대로 유효**. 미디어 자산·완성본의 상태는 별도 media 도메인 없이 `content` 도메인이 소유(무상태 워커 + 상태 소유 분리).
- **대안(기각)**: 로컬 GPU 미디어 스택(Stable Diffusion 등 자체 호스팅) — 하드웨어·운영 비용 과다, 단일 사용자 규모에 과함.
- **관련**: [0002-pgvector-rag], [0003-external-media-api-guardrail]. CLAUDE.md §1·§4, `doc/ref/architecture/README.md` 반영.
