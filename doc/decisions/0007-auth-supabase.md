# ADR 0007 — 인증을 Supabase Auth로 (원형 자체 JWT 대체)

- **상태**: 채택 (2026-07-19)
- **맥락**: 원형(py-msa-ai)은 `auth` 서비스가 JWT를 자체 발급하고, 게이트웨이가 `JWT_SECRET`으로 검증한다. 사용자 관리·토큰 발급·비밀번호 정책을 직접 운영해야 한다. 인증은 알파가 아니라 **커모디티** — 자체 구현·보안 유지보수는 알파 아닌 곳에 드는 비용이다.
- **결정**: 인증을 **Supabase Auth**로 위임한다. 스코프는 **인증만(A)** — DB·Storage는 기존 자체 Postgres를 그대로 쓴다.
  - Supabase가 **사용자·세션·JWT 발급**을 소유.
  - 게이트웨이는 Supabase가 발급한 JWT를 **JWKS(공개키)로 검증** → `UserContext` 생성 → 하류에 **HMAC 신뢰헤더**(원형 패턴 유지).
  - 원형 `auth` 서비스 = **제거 또는 얇은 프로필 래퍼**(도메인 프로필 확장이 필요할 때만). `auth_db` = Supabase가 계정을 관리하므로 기본 불필요.
- **스코프 제외(기각)**: (B) Supabase Postgres·(C) Supabase Storage. 서비스별 자체 Postgres와 미디어 저장은 별도 결정으로 남긴다. **인증만 위임**해 외부 결합을 최소화.
- **트레이드오프**: 외부 관리형 의존성 1개 추가(계정 데이터가 Supabase에 상주) vs 인증 구현·보안 유지보수 제거. 인증은 커모디티라 위임이 맞다. 리서치 원문·텍스트 추론은 여전히 로컬(가드레일 불변).
- **가드레일 정합**: 텍스트 LLM·원문 로컬 유지 **불변**. 콘텐츠 자동발행 금지 **불변**. 신규: 인증 PII는 Supabase에 상주 — **리서치 원문과 분리**(원문은 여전히 외부 반출 금지).
- **영향**: 게이트웨이 JWT 검증 로직 교체(JWKS), `auth` 서비스 정리, `.env`에 `SUPABASE_*` 키 추가, `docker-compose` `auth` 조정, `doc/design/README` 로드맵에 **라운드 0(인증)** 추가.
- **관련**: [0006]. 원형 게이트웨이 인증 패턴. 반영: `CLAUDE.md` §1, `doc/design/README.md`(라운드 0), `doc/ref/architecture/`.
