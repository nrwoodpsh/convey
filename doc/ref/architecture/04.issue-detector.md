# 04. issue-detector 아키텍처 (알파 2)

"오늘 뭘 만들지"를 자동으로 포착하는 컴포넌트. **도메인이 아니라 서비스(워커)** — research가 쌓은 신호를 읽어 **이슈 종목 랭킹**을 만들고, 콘텐츠 제작의 실질 방아쇠를 제공한다. 편집적 신호가 알파.

## 구성

| 서비스 | 배포형태 | 역할 |
|:---|:---|:---|
| `issue-detector` | 워커 | `market.ticks` + `research.ingested` 구독 → 시세 급변 + 뉴스 빈도 집계 → 이슈 종목 랭킹 |

## 흐름

```
market.ticks (시세)  ─┐
research.ingested (뉴스 빈도) ─┤
                              ▼
[issue-detector]  종목별 신호 집계 (급등락·거래량·뉴스 volume)
                  → "오늘의 이슈 종목" 랭킹
                  ▼
        사람 확인/선택 → POST /content/generate   (자동 큐잉은 후속)
```

## 설계 포인트 (/design 확정)

- **DB 경계**: research_db에 직접 붙지 않음. **Kafka 스트림 구독**으로 자체 신호 집계(윈도우/자체 소형 상태). 서비스별 DB 원칙 유지.
- **랭킹 로직**: 시세 변동성·거래량 급증 + 뉴스 빈도 가중. 구체 지표·윈도우는 설계.
- **출력**: 이슈 종목 랭킹을 조회(API) 또는 이벤트(`issue.detected`)로 노출 — 방식은 설계.
- **사람 개입**: 초기엔 랭킹을 사람이 보고 제작 지시(자동 큐잉은 양산 라운드 ⑤).

## 이벤트

- **구독**: `market.ticks`, `research.ingested`
- **발행(안)**: `issue.detected` 또는 조회 API — /design 확정

## 빌드 순서상 위치

빌드 ② (데이터 기반 ① 다음). research 데이터가 있어야 랭킹이 성립.
