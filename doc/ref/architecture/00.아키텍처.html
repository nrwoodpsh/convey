03

데이터 · 지식 저장

LoRA를 안 쓰므로 지식 레이어가 품질의 유일한 레버. 관계·인과 = 그래프(알파), 사실 = Postgres. research가 두 스토어 소유(ADR 0005).

Neo4j · research_graph

관계·인과 — 알파① 엔진 (다홉 traversal)

  • 노드: Stock · Event · Sector · Company
  • 엣지: HAS_EVENT · AFFECTS · SUPPLIES · COMPETES · BELONGS_TO
  • 용도: "삼성 실적↑ → 협력사 수혜 → 섹터 파급" 관계 사슬

Postgres · research_db +pgvector

사실 — SQL 정확 조회 + 의미검색 보조

  • PriceTick 시세 시계열 (차트·이슈감지)
  • Article 원문 + 출처·라이선스 (필수)
  • Embedding pgvector (테마 의미검색 보조)
삼성전자 · StockHAS_EVENT실적발표 · Event
삼성전자 · StockAFFECTS · 수혜협력사 A · StockSUPPLIES삼성전자 · Stock
삼성전자 · StockBELONGS_TO반도체 · Sector

진짜 일 = 추출 파이프라인(뉴스 → NER + 관계추출 → 그래프 엣지). 여기가 품질의 원천. news-feed 태깅 + research 관계 구축.

04

이벤트 · 토픽

단계 결합은 Kafka 비동기 잡. 축적과 제작은 자동으로 안 이어짐 — 사람의 이슈 종목 선택이 방아쇠.

토픽발행구독비고
market.ticksmarket-feedresearch · issue-detector시세 스트림
research.ingestednews-feedresearch · issue-detector축적 (콘텐츠 자동유발 X)
content.generatecontent (API·on-demand)content consumer잡 큐 (비동기)
image.generate · tts.generatecontentimage-gen · tts (외주)미디어 fan-out
content.approved사람 승인publishing발행 트리거
content.publishedpublishing(관측)발행 완료

이슈 종목 랭킹·합성 완료는 이벤트가 아니라 내부 상태/조회. 토픽명은 /design 확정.

05

도메인 상세

3도메인(research·content·publishing) + issue-detector 컴포넌트. 각 서비스·저장·이벤트·핵심.

① research — 축적

서비스research(API) · market-feed · news-feed(워커)
저장Neo4j 그래프 + Postgres research_db+pgvector
이벤트발행 market.ticks·research.ingested
핵심종목·사건 관계 축적. 출처·라이선스 필수. 자동 유발 X

② issue-detector — 선별

서비스issue-detector(워커)
저장스트림 자체 집계 (DB 직접접속 X)
이벤트구독 market.ticks·research.ingested
핵심시세급변+뉴스빈도 → 오늘의 이슈 종목 랭킹 (알파2)

③ content — 제작

서비스content·agent·llm-inference·image-gen·tts·video-assembly
저장Postgres content_db+pgvector(히스토리)
이벤트자기큐 content.generate / fan-out 미디어 / 발행 content.approved
핵심근거 스크립트 + 정확 렌더. 미디어 자산 소유. 승인 후 발행

④ publishing — 발행

서비스publishing(API)
저장Postgres publishing_db
이벤트구독 content.approved / 발행 content.published
핵심YouTube 업로드(커모디티). 승인 없이 업로드 금지
06

크로스커팅 제약

원형(py-msa-ai) 계승 + 정확성을 지키기 위한 규칙.

단일 진입 · HMAC

외부 요청은 gateway(:8080)만. 서비스 간 직접 호출은 HMAC 신뢰헤더로만.

서비스별 DB

DB 공유 금지. 데이터는 API·Kafka로만. issue-detector도 스트림 구독(직접 접속 X).

Kafka 비동기 잡

단계 결합은 이벤트. 조회성만 동기 API. 아웃박스(커밋 후 발행).

지식 = 그래프 + 사실 + 벡터

관계·인과 = Neo4j 그래프(다홉 추론, 알파). 시세·기사 = Postgres SQL. 의미검색 = pgvector 보조.

east-west 직결

agent → research/content /search · llm-inference는 gateway 재진입 없이 HMAC 직결.

로컬 vs 외부

텍스트·임베딩·차트 렌더 = 로컬(Ollama/ffmpeg). broll·TTS·업로드만 외부 API.
